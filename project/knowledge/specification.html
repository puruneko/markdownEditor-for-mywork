MD AST Editor

バージョン: 0.0.1 最終更新: 2026-06-26 対象: Obsidian プラグイン（デスクトップ専用）

プロジェクト概要

MD AST Editor は Obsidian 上で動作するタスク管理プラグインです。 Markdown ファイルをリアルタイムで解析し、タスクを Calendar・Gantt・Kanban の3種類のビューで視覚化します。 ビュー上での操作（ドラッグ＆ドロップ、ステータス変更等）は即座に元の Markdown ファイルに書き戻されます。

主要機能

  • Markdown ファイルのリアルタイムパースと AST 生成
  • Calendar ビュー：スケジュール付きタスクをカレンダー形式で表示・編集
  • Gantt ビュー：タスクの期間と依存関係をガントチャートで表示
  • Kanban ビュー：ステータスとセクション階層でカード分類・ドラッグ移動
  • AST デバッグビュー：パース結果の JSON を表示
  • CodeMirror タスクハイライト：エディタ上でステータスをカラー装飾
  • ビュー上のノードクリックでエディタカーソルを対応行に移動

設計方針：データストアは Markdown ファイルのみ。専用 DB や独自フォーマットは使用しない。 タスクのスケジュールや優先度はインラインのメタフィールド（@schedule: 等）として記述する。

レイヤー構成は以下の通りです。上位レイヤーが下位レイヤーに依存し、逆依存はありません。

↓

↓

タスク記法

タスクは GFM（GitHub Flavored Markdown）のチェックボックス形式をベースに、カスタムマーカーを追加した独自記法を使います。

ステータスマーカー

- [ ] タスク todo
- [x] タスク done
- [>] タスク doing
- [!] タスク blocked
- [-] タスク hold
- [X] タスク done（大文字 X も可）
マーカーステータス説明
[ ]todoGFM 標準チェックボックス（未チェック）
[x] / [X]doneGFM 標準チェックボックス（チェック済み）
[>]doingカスタムマーカー（作業中）
[!]blockedカスタムマーカー（ブロック中）
[-]holdカスタムマーカー（保留）

非タスクリスト（ListNode）

チェックボックスのないリスト項目はタスクではなく リストノード として扱われます。 Kanban ビューではグルーピングの境界として機能します。

- グループ名（チェックボックスなし → ListNode）
  - [ ] このタスクは「グループ名」セクション内に属する

ネスト構造

タスクは任意の深さでネストできます。子タスクは親タスクのインデント配下に記述します。

- [ ] 親タスク
  - [>] 子タスク（作業中）
    - [!] 孫タスク（ブロック中）
  - @schedule: 2026-07-01T10:00/2026-07-01T12:00  ← 親タスクのメタ

注意：インデントはタブ文字を使用してください（Obsidian のデフォルト設定 "Use tabs" に合わせています）。 スペースインデントも許容されますが、タブに正規化されてシリアライズされます。

メタフィールド

タスクの追加情報はタスク行の子リスト内に @key: value 形式で記述します。 パース時にメタフィールドは AST から除去され、タスクノードの meta プロパティに集約されます。

- [ ] タスク名
  - @schedule: 2026-07-01T10:00/2026-07-01T12:00
  - @due: 2026-07-05
  - @priority: 1
  - @tags: frontend, urgent
  - @dependsOn: task-id-abc, task-id-def
キー型説明例
@schedule string（ISO 日時範囲） 作業予定期間。Calendar・Gantt ビューで使用。 2026-07-01T10:00/2026-07-01T12:00
@due string（ISO 日付） 期限日。Kanban カードに表示。 2026-07-05
@priority number 優先度（数値が小さいほど高優先）。Kanban カードに表示。 1
@tags string[]（カンマ区切り） タグ一覧。フィルタリング用。 frontend, urgent
@dependsOn string[]（カンマ区切り） 依存タスクの ID。将来の依存関係表示に使用予定。 task-id-abc

実装詳細：メタフィールドは remark-meta-fields プラグインが処理します。 タスクノードの子リストを走査し @key: value パターンにマッチした行を収集・削除します。 シリアライズ時はタブインデントで再出力されます。

スケジュール書式

@schedule の値は ISO 8601 の日時範囲形式で記述します。 パーサーは省略記法を自動展開します（schedule-normalize.ts）。

正規化ルール（3段階）

段階処理入力例展開後
1. 継続展開 終了側に省略された日付を補完 26-01-01T10:00/11:30 26-01-01T10:00/26-01-01T11:30
1. 継続展開（日） 終了側に省略された年月を補完 26-01-01/02 26-01-01/26-01-02
2. 2桁年号展開 YY- を 20YY- に展開 26-01-01 2026-01-01
3. 分省略展開 THH を THH:00 に展開 2026-01-01T10 2026-01-01T10:00

書式パターン例

# 最も省略した書き方
- @schedule: 26-07-01T10/12    → 2026-07-01T10:00/2026-07-01T12:00

# 日付のみ（終日イベント）
- @schedule: 2026-07-01/2026-07-03

# 時刻付き（同日）
- @schedule: 2026-07-01T10:00/2026-07-01T12:00

# 時刻付き（複数日）→ Calendar では dateRange として扱う
- @schedule: 2026-07-01T09:00/2026-07-03T18:00

Calendar ビューでの解釈

種別条件Calendar 表示
dateTimeRange同日・時刻付き開始〜終了時刻の時間ブロック
dateRange日付のみ or 複数日・時刻付き日付またがりの全日イベント

セクション構造

Markdown の Heading（# 〜 ######）がセクション境界になります。 セクションは Heading レベルに基づいて自動的にネストされます。

# プロジェクトA （depth=1 Section）

- [ ] 直接タスク

## フェーズ1 （depth=2 Section、プロジェクトAのサブセクション）

- 機能実装 （ListNode = グルーピング用）
  - [>] API 設計         （section: ["フェーズ1", "機能実装"]）
  - [ ] DB スキーマ設計   （section: ["フェーズ1", "機能実装"]）

- テスト （ListNode）
  - [ ] ユニットテスト作成  （section: ["フェーズ1", "テスト"]）

無名セクション：最初の Heading より前にコンテンツがある場合、id: 'section-0'・lineNumber: -1 の無名セクションとして扱われます。

パーサー処理パイプライン

エントリーポイントは src/lib/parser/parse-markdown.ts の parseMarkdown(markdown: string): Document 関数です。

① stripCommonIndent()
タブ/スペース混在インデントを正規化。Obsidian のタブインデントによる誤解析を防止。
↓
② remark-parse + remark-gfm
Markdown → mdast.Root を生成。GFM テーブル・チェックボックス対応。
↓
③ remark-task-status プラグイン
ListItem を走査し、checked フラグおよびカスタムマーカー（[>][!][-]）から Status を決定。node.data.taskStatus に格納。
↓
④ remark-meta-fields プラグイン
タスク配下のリストから @key: value 行を検出・収集し node.data.meta に格納。該当 ListItem を mdast から削除。
↓
⑤ buildSectionsFromRoot()
mdast を走査し Heading でセクションを区切り、nestSections() で深度ベースの階層を構築。buildNodeLineMap() で ID→行番号マップを生成。
↓
⑥ Document 出力
{ sections: Section[], nodeLineMap: Map<string, number> } を返す。

ID 生成（generateId）

各ノードには path: string[]（セクション → ノードへのテキスト経路）の DJB2 ハッシュから決定論的な ID が生成されます。 同じ Markdown を再パースすると同じ ID が生成されるため、ビューとの紐付けが安定します。

AST データモデル

型定義はすべて src/lib/parser/types.ts に集約されています。

Document（ルート）

interface Document {
  type: 'document'
  sections: Section[]
  nodeLineMap: Map<string, number>  // id → 0-based 行番号（エディタカーソル移動用）
}

Section（セクション）

interface Section {
  type:           'section'
  id:            string           // 決定論的ハッシュ ID
  depth:         number           // Heading レベル（1〜6）
  title:         string           // Heading テキスト
  lineNumber:    number           // 0-based 行番号（-1 = 無名セクション）
  parentSectionId?: string
  children:     Node[]            // Section 直下のノード
  subSections:  Section[]         // ネストされたサブセクション
}

Node 共用体型

type Node = TaskNode | ListNode | QuoteNode
type Status = 'todo' | 'doing' | 'done' | 'blocked' | 'hold'

TaskNode（タスク）

interface TaskNode {
  type:    'task'
  id:      string
  text:    string    // タスクタイトル（マーカー除く）
  status:  Status
  children: Node[]   // ネスト子ノード
  meta?: {
    schedule?:  string    // ISO 日時範囲
    due?:       string    // ISO 日付
    priority?:  number
    dependsOn?: string[]
    tags?:      string[]
  }
  lineNumber:        number   // 0-based
  depth:             number   // ネスト深さ
  path:              string[] // セクション→ノードの経路（ID 生成に使用）
  hasTaskDescendant: boolean
  isGroup:           boolean  // hasTaskDescendant && children.length > 0
  isLeafTask:        boolean  // children.length === 0
  isMemo:            false
}

ListNode（非タスクリスト）

interface ListNode {
  type:    'list'
  id:      string
  text:    string
  children: Node[]
  lineNumber:        number
  depth:             number
  path:              string[]
  hasTaskDescendant: boolean
  isGroup:           boolean  // Kanban グルーピング境界として機能
  isMemo:            boolean  // !hasTaskDescendant の場合 true
}

QuoteNode（引用ブロック）

interface QuoteNode {
  type:              'quote'
  raw:               string   // 引用テキスト（改行含む）
  hasTaskDescendant: false
  isGroup:           false
  isMemo:            true
}

シリアライザ（AST → Markdown）

src/lib/parser/ast-to-md.ts の serializeAst(doc: Document): string が Document を Markdown 文字列に逆変換します。 ビュー上での操作後は直接 markdown-patch.ts の行差分パッチが使われるため、 シリアライザはエディタ上での手動呼び出しや テストで使用されます。

出力ルール

ノード型出力形式
Section (depth=1)# タイトル
Section (depth=2)## タイトル
TaskNode (todo)- [ ] テキスト
TaskNode (doing)- [>] テキスト
TaskNode (done)- [x] テキスト
TaskNode (blocked)- [!] テキスト
TaskNode (hold)- [-] テキスト
ListNode- テキスト
QuoteNode> テキスト
メタフィールド（子）TAB- @key: value

Markdown 行差分パッチ（markdown-patch.ts）

ビュー操作（カード移動・日程変更等）は AST 全体を再生成せず、 対象の 特定行のみを差分パッチします。これにより手動編集済みのコメント・空行・カスタムフォーマットが保持されます。

関数用途検索戦略
patchNodeStatus() ステータス変更（Kanban カード移動） node.lineNumber で直接行指定（最も正確）
patchScheduleForNode() スケジュール変更（Calendar ドラッグ） タスク本文行を検索 → インデント配下の @schedule 行を更新
patchTaskTitle() タスク名変更 タスク行パターンマッチ

Calendar ビュー

@schedule を持つタスクをカレンダー形式で表示します。 UI コンポーネントは svelte-calendar-lib（ローカルライブラリ）を使用します。

データ変換（ast-to-calendar.ts）

type CalendarItem = {
  id:          string
  title:       string
  taskStatus:  'todo' | 'doing' | 'done'   // blocked/hold → todo に集約
  startDate:   DateTime
  endDate:     DateTime
  parentTitle?: string
}
  • @schedule を持たないタスクはカレンダーに表示されません
  • ステータスは todo / doing / done の3値に集約されます（blocked / hold → todo）
  • カレンダー上のイベントをクリックするとエディタの対応行にカーソルが移動します
  • ドラッグ＆ドロップで @schedule の値を書き換えます（patchScheduleForNode()）

ファイル役割
src/views/CalendarView.tsObsidian ItemView サブクラス（Shadow DOM）
src/views/CalendarViewMount.svelteSvelte マウントポイント・FileSync 購読
src/lib/calendar/CalendarTab.svelteタブ UI（svelte-calendar-lib をラップ）
src/lib/calendar/ast-to-calendar.tsAST → CalendarItem 変換

Gantt ビュー

タスクとその階層構造をガントチャートで表示します。 UI コンポーネントは svelte-gantt-lib（ローカルライブラリ）を使用します。

データ変換（ast-to-gantt.ts）

  • @schedule を持つタスクのみ対象
  • 子孫に @schedule を持つグループノードは自動的に親ノードとして Gantt に追加
  • descendantDateRange() で子孫の min/max スケジュールから親の期間を推定
type GanttNodeType = 'project' | 'section' | 'subsection' | 'task'

// 判定ロジック
if (hasSchedule && !hasChildSchedule) → type: 'task'
if (!hasSchedule && hasChildSchedule) → type: 'subsection'（子孫範囲から期間計算）

Kanban ビュー

タスクをステータスレーン（Todo / Doing / Blocked / Hold / Done）と セクション階層グループで分類して表示します。 UI コンポーネントは svelte-kanban-lib（ローカルライブラリ）を使用します。

KanbanCard 型

type KanbanCard = CardData & {
  id:           string
  title:        string
  status:       string     // 'todo' | 'doing' | 'done' | 'blocked' | 'hold'
  section:      string[]   // セクション階層パス（ライブラリの groupBy で使用）
  sectionTitle: string     // section[0]（後方互換）
  groupTitle:   string     // section の末尾（後方互換）
  depth:        number
  schedule?:   string
  due?:         string
  priority?:   number
  tags?:        string[]
}

section フィールドとグルーピング

section: string[] は Markdown 構造から自動生成されます。 ライブラリの groupBy: 'section' + sectionDepth: 2 設定により、 ライブラリが配列を "Heading / List" 形式の groupId に変換してグループヘッダーを自動生成します。

# プロジェクト

- 機能A（ListNode）
  - [ ] タスク1    → section: ["プロジェクト", "機能A"]
  - [>] タスク2    → section: ["プロジェクト", "機能A"]

- 機能B（ListNode）
  - [ ] タスク3    → section: ["プロジェクト", "機能B"]

# Kanban 表示（グループモード）
┌─────────────────────────────────────────┐
│ プロジェクト                              │ ← groupPath（小さく表示）
│ 機能A                                    │ ← groupLabel（大きく表示）
│  [Todo] タスク1  [Doing] タスク2         │ ← レーンが横並び
├─────────────────────────────────────────┤
│ プロジェクト
│ 機能B
│  [Todo] タスク3
└─────────────────────────────────────────┘

開発ワークフロー

新機能開発の手順

  1. Issue 作成：project/issues/obs/obs-XXXX-feature-name.md を作成。 実装は必ず Issue 番号に紐づける（Issue なし実装は禁止）。
  2. ローカルライブラリ更新（必要な場合）：npm run libs:update
  3. 実装：TypeScript / Svelte 5 でコーディング
  4. テスト実行：npm run test:unit（全 183 テストパスを維持）
  5. ビルド確認：npm run build:dev（Obsidian に自動コピー）
  6. Obsidian 動作確認：実際の Obsidian でビューを確認
  7. Issue クローズ承認：ユーザーの明示的な承認を得てから Issue をクローズ
  8. コミット：Issue クローズ後にコミット

ガバナンスルール： project/__private/ および *.__private.md ファイルは AI からアクセス禁止。 Issue クローズはユーザーの明示的承認が必要。コミットは Issue クローズ後のみ。

Shadow DOM とスタイリング

すべての Obsidian ビューは ShadowItemView が生成する Shadow DOM ホスト内でレンダリングされます。 これにより Obsidian の CSS との衝突を防いでいます。

Kanban ビューは CSS カスタムプロパティで Obsidian のデザイントークンを参照します：

--kanban-bg: var(--background-primary, #1e1e1e);
--kanban-accent: var(--interactive-accent, #4ec9b0);
