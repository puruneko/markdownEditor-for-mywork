1. エグゼクティブサマリー

本プロダクトは 「素の Markdown を唯一の正（single source of truth）とし、そこから AST を介してカレンダー・ガント・カンバンへ双方向同期する」 という設計思想を持つ Obsidian プラグインである。unified/remark ベースの堅牢なパーサーと、ドラッグ操作を Markdown 本文へ書き戻す双方向同期は、既存の Obsidian 系タスクプラグイン群と比べても明確な独自性・優位性がある。

一方で、現状は 「単一ファイルを 4 つのビューで見る」段階 に留まっており、実用的なタスク管理ツールとして広く使われている機能群 ―― ① 複数ファイル横断の集約・クエリ、② 繰り返しタスク、③ 今日／期限のアジェンダ、④ リマインド通知、⑤ フィルタ・検索 ―― が未実装である。これらは競合（Obsidian Tasks / Dataview / Projects、org-agenda 等）が標準で備える領域であり、ロードマップの中核に据えるべきギャップである。

結論： 「記法・パーサー・双方向ビュー」という土台は完成度が高い。次フェーズは 土台の上に載る “使うための機能”（集約・繰り返し・アジェンダ・通知） へ投資するのが費用対効果が高い。

2. 現状の機能棚卸し

コードベースから確認できる実装済み機能を層別に整理する。

コア：記法 & パーサー

  • unified/remark + GFM ベースのパーサー（parse-markdown.ts）
  • 独自 remark プラグイン：remark-task-status / remark-meta-fields
  • AST：Section / TaskNode / ListNode / QuoteNode / Document
  • 派生プロパティ：isGroup / isMemo / hasTaskDescendant / depth / path
  • ステータス 5 種：[ ] [x] [>] [!] [-]（todo/done/doing/blocked/hold）
  • メタ（子リスト記法）：@schedule @due @priority @dependsOn @tags
  • 日付省略記法・タブ/2スペースインデント吸収・nodeLineMap（行番号対応）

ビュー（4種・自作 svelte ライブラリ）

  • AST View — 構造ツリー表示
  • Calendar View — @schedule を月/週で表示
  • Gantt View — スケジュール範囲をバー表示
  • Kanban View — セクション単位でグルーピング、記述順を保持
  • いずれも ../calendar-for-mywork 等の自作ライブラリを file: 参照

同期・連携

  • MD ↔ AST リアルタイム同期（デバウンス制御）
  • カレンダー/ガントの ドラッグ→Markdown 本文へ書き戻し（markdown-patch.ts）
  • 編集ダイアログ→Markdown 反映
  • カード/イベントのクリック→エディタカーソルへジャンプ（スクロールオフセット設定可）
  • EditorEventBus による疎結合なビュー間連携

エディタ / 設定 / 品質

  • CodeMirror6 装飾：ステータス・メタのハイライト（task-decoration.ts）
  • 設定：リボン表示 / ハイライト ON-OFF / デバウンス ms / スクロール行数
  • テスト：vitest（単体）/ playwright（E2E）/ wdio-obsidian（Obsidian E2E）
  • 厳格なガバナンス（Issue 駆動・トレーサビリティ・言語ポリシー）

3. 本プロダクトの強み（競合優位）

① 素の Markdown が唯一の正

専用 DB やフロントマター・別ファイル形式に閉じ込めず、人が読めて Git 差分も取れる平文をそのまま編集対象にする。可搬性・耐ロックイン性が高い。

② 双方向・ドラッグ書き戻し

カレンダー/ガントでのドラッグ結果を Markdown 本文へ書き戻す。多くの Obsidian カレンダー系プラグインは「表示専用」または frontmatter 限定で、本文インラインタスクへの書き戻しは希少。

③ remark ベースの正統な AST

正規表現の場当たりではなく unified パイプライン上の AST。記法拡張やビュー追加に強く、テストで仕様を固定できている。

④ 開始/終了レンジの @schedule

単なる due 日付ではなく開始〜終了の時間幅を一級で扱う。ガント/時間ブロッキングに直結する。同一タスク複数 schedule も可。

⑤ 1 ソース → 4 ビュー

同じ Markdown を AST/カレンダー/ガント/カンバンで同時に見られる。Projects プラグインに近いが、こちらはインラインタスクが対象。

⑥ メモ・引用の一級扱い＆カーソル同期

QuoteNode（経緯メモ）やグループ/メモを構造として保持。ビュー→エディタのカーソルジャンプで「見て→直す」が速い。

4. 不足機能・改善候補（実装方針つき）

MUST 実用に不可欠／競合が標準装備 SHOULD 体験を大きく底上げ COULD 差別化・将来投資

4.1 データ集約・スコープ — 最重要カテゴリ

MUST  複数ファイル／Vault 横断の集約

課題 FileSync が「現在開いているファイル」1 枚しか追跡しない。実運用ではタスクは複数ノートに散らばるため、横断ビューが作れないのが最大の制約。Obsidian Tasks / Dataview が最も評価される機能がこれ。

実装方針： FileSync を「単一ファイル追跡」から「ソース集合」へ拡張。① app.vault.getMarkdownFiles() を起動時に走査し各ファイルを parse → AST をメモリ index 化、② metadataCache／vault.on('modify') で差分更新、③ 各ノードに sourcePath を付与（書き戻し先の特定に必須）。スコープは「現在ファイル / 指定フォルダ / タグ / Vault 全体」を設定で選択。nodeLineMap を (path,line) キーへ拡張する。

MUST  クエリ／フィルタ言語と保存フィルタ

課題 「未完で今週期限」「#総務 かつ [!]」のような絞り込み手段が無い。集約と対で必要。

実装方針： AST に対する宣言的フィルタ DSL を定義（例：status:todo,doing due:<=2026-07-05 tag:総務）。まずは構造化オブジェクト（{status, dueBefore, tags, sectionPath, priority}）として実装し、各 ast-to-* 変換の前段に filterNodes(doc, query) を挟む。フィルタはコードブロック（```task-query）として Markdown に埋め込み or ビュー上の UI から指定。よく使う条件は設定に保存（スマートリスト）。

4.2 時間・繰り返し

MUST  繰り返しタスク（RRULE）

課題 project/__private/workspace/knowledge/RRULE.md に検討メモがあるが未実装。「毎月第2・第4金曜の清掃」のような定期業務（ガイドの実例そのもの）が表現できない。org-agenda / Tasks / Todoist は標準装備。

実装方針： メタに @repeat（または @rrule）を追加 → meta-keys.ts と remark-meta-fields を拡張、Meta 型に repeat?: string。値は RFC 5545 RRULE（FREQ=WEEKLY;BYDAY=FR;INTERVAL=2）または簡易記法（every 2 weeks on Fri）。rrule ライブラリで @schedule を起点に表示範囲分のオカレンスへ展開し、カレンダー/ガントへ投入。完了時の挙動（次回生成 or チェック蓄積）を仕様決定する。展開はビュー側のみ（本文は 1 行のまま）に留めると書き戻し問題を回避できる。

SHOULD  カレンダーの日/週/アジェンダ切替・終日帯

課題 __private の prompt に week/month/day 検討が残るが、ビュー切替が UI として確立していない。@due のみ（時刻なし）のタスクが終日扱いで一覧化されると有用。

実装方針： ast-to-calendar.ts に @due を all-day イベントとして射影するパスを追加（現状は @schedule 中心）。ビュー側ライブラリ（calendar-for-mywork）に view mode（month/week/day/agenda）を切替プロップとして渡す。

4.3 アジェンダ・通知

MUST  Today / アジェンダビュー（期限ダッシュボード）

課題 「今日やること」「期限超過」を集約表示する画面が無い。タスク管理ツールで最も日常的に開かれる画面。org-agenda の中核。

実装方針： 4.1 の集約基盤の上に新ビュー AgendaView を追加。@due/@schedule を「期限超過 / 今日 / 今週 / 予定なし」へ束ねて時系列リスト表示。各行クリックで既存のカーソルジャンプを再利用。優先度・ステータスでソート。

SHOULD  リマインド通知

課題 期限・開始時刻が来ても何も起きない。

実装方針： プラグイン常駐タイマーで直近の @schedule 開始/@due を監視し、Obsidian Notice（＋デスクトップは Notification API）で通知。デフォルトのリードタイム（例：開始10分前）を設定化。@reminder メタで個別指定も可。

4.4 進捗・依存・可視化

SHOULD  グループの進捗ロールアップ

グループ（isGroup）配下の完了率（3/5・60%）が出ない。

方針： AST 派生段で子孫タスクの done 比率を集計し progress として付与。カンバン/ガント/AST のグループ見出しにバー表示。純粋な派生プロパティなので書き戻し不要・低リスク。

SHOULD  @dependsOn の依存矢印（ガント）

dependsOn はモデルにあるが ast-to-gantt.ts で可視化されていない。ID 解決もガイド上曖昧。

方針： まず安定した参照 ID（@id メタ or 行ベース slug）を確立し、ガントへ依存リンク（dependency）を渡して矢印描画。循環検出と未解決 ID の警告も。

COULD  色分け（タグ/優先度/ステータス）

全ビューで一貫した色規則が無い。

方針： 設定で「タグ→色」「優先度→色」マップを定義し、ast-to-* 出力の color に反映。凡例表示。

COULD  カンバンのドラッグ＝ステータス遷移

現状グルーピングはセクション。列移動でステータス（todo→doing→done）を変える運用も一般的。

方針： groupByField に status を選べるようにし、列間ドラッグ時に [ ]→[>]→[x] を Markdown へ書き戻し（既存 patch 基盤を流用）。

4.5 入力・編集体験

SHOULD  クイック追加 & 日付ピッカー

タスク追加・日付入力が手打ち。記法ミス（@schedule 形式）が起きやすいことはガイドの「よくあるミス」表が示す。

方針： コマンド＋モーダルで「タスク文＋日付＋ステータス」を入力→正規形を挿入。日付は Obsidian の suggest／カレンダーポップオーバー。自然言語入力（「明日10時」）は chrono-node で補助。

COULD  ステータス・メタキーのカスタマイズ

5 ステータス・5 メタキーがハードコード。組織ごとに語彙が違う。

方針： META_KEYS とステータスマップを設定駆動に。ただし記法の安定性とトレードオフのため、まずは別名（alias）対応から。

4.6 連携・堅牢性

COULD  iCal / 外部カレンダー連携（書き出し）

予定を Google Calendar 等で見たい需要。

方針： @schedule 群を .ics へエクスポートするコマンド。双方向は競合多発のため当面エクスポートのみ。

SHOULD  外部編集・競合への耐性 / モバイル検証

書き戻し中に本文が外部更新された場合の整合、巨大ファイルでのデバウンス負荷、モバイル（CM6 API 依存箇所）未検証。

方針： 書き戻しを「行範囲の最小パッチ＋再 parse 検証」にし、行ズレ時は中断して通知。(mdView.editor as any).cm 依存（plugin.ts のスクロール処理）にモバイルフォールバックを用意。

5.2 製品別：本アプリが「劣る」ポイントと採用候補

各カードは「その製品に本アプリが負けている具体機能」→「だから何を取り入れるか」をセットで示す。

Obsidian TasksObsidian プラグイン・インラインタスクのデファクト

負けている点： ① ```tasks クエリブロックで Vault 全体のタスクをノート内に集約表示できる ② 繰り返し（🔁 every week）③ 「タスク作成/編集」モーダル（日付ピッカー・ステータス・優先度を GUI で入力し、記法は自動生成）④ 入力中のオートサジェスト ⑤ 相対日付（"in two weeks"）。本アプリはこれらが すべて未実装。

取り入れるべき： 採用候補 (a) 埋め込みクエリブロック（```task-query をノートにレンダリング）、(b) 繰り返し記法、(c) 編集モーダル＋日付ピッカー。特に (c) は「@schedule 形式を手打ちさせない」＝本アプリの記法ミス（ガイドの注意点表）を構造的に撲滅できる。
Dataview / Datacoreクエリ・集約のデファクト

負けている点： ノートに ライブクエリを埋め込み、表示時に最新のタスク一覧へ自動展開（TABLE / LIST / TASK）。「管理簿を手で更新しない」運用の核。本アプリのビューは 外部パネルで、ノート内に「この案件の未完タスク一覧」を差し込むことができない。

取り入れるべき： 採用候補 ノート埋め込み型のライブ集約ブロック。例：案件ノートに ```task-query scope:folder status:!done を書くと、その場に最新一覧が描画される。転記レスの決定打。
Projects（Marcus Olsson）多ビュー・ノートDB

負けている点： 編集可能なテーブル（表）ビュー。スプレッドシートのようにセルを直接編集して一括で日付・ステータスを直せる。本アプリには 表ビューが無い（AST/カレンダー/ガント/カンバンのみ）。大量タスクの棚卸し・一括編集で大きく劣る。

取り入れるべき： 採用候補 編集可能テーブルビュー（列：タスク/ステータス/期限/スケジュール/優先度/タグ/出典ファイル）。セル編集は既存の Markdown 書き戻し基盤に接続。一覧で見て、その場で直す＝転記レス＋抜け漏れ点検に効く。
Day Plannerタイムブロッキング

負けている点： 1日のタイムラインに予定を縦に並べ、未定タスクをドラッグして時間に割り付け、現在時刻インジケータで「今やること」を示す。本アプリのカレンダーは月/週中心で、1日の時間設計（タイムブロッキング）UI が弱い。

取り入れるべき： 採用候補 デイ・タイムラインビュー。未スケジュールのタスクを今日の時間帯にドラッグ → @schedule を書き戻す。「予定に落とし込めていないタスク」を可視化し抜け漏れを防ぐ。
org-mode / org-agenda（Emacs）平文タスク管理の到達点

負けている点： ① 横断アジェンダ（全ファイルの SCHEDULED/DEADLINE を日付順に集約）② キャプチャ（どこからでも素早く取り込み、定位置へ）③ Refile（タスクを別ファイル/見出しへ移動し参照を維持）④ clocking（作業時間の記録）。本アプリは①〜④いずれも未実装。

取り入れるべき： 採用候補 アジェンダ・クイックキャプチャ・Refile（タスク移動）。Refile は「散在した管理簿を1か所へ寄せる」哲学に直結する。clocking は優先度低。
Todoist / TickTick / Things 3SaaS タスク管理（UX 参照）

負けている点： ① 自然言語クイック追加（「請求書 明日17時 #経理 p1」を1行で構造化）② スマートリスト（今日 / 次の予定 / いつか）③ ubiquitous capture（常時どこからでも追加）④ 通知 ⑤ TickTick は習慣・ポモドーロも内包。平文ではないが 「入れる速さ」「拾い上げる確実さ」で大きく勝る。

取り入れるべき： 採用候補 自然言語クイック追加・常設スマートリスト（今日/期限切れ/未スケジュール）。速く入れられないと結局メモ帳に転記され、散在・抜け漏れが起きるため哲学に直結。
obsidian-reminder / Logseq通知・日次運用

負けている点： obsidian-reminder は 期限の通知と 日付入力サジェストを提供。Logseq は 日次ノート（ジャーナル）中心の捕捉と LATER/NOW 運用。本アプリは通知ゼロ・デイリーノート連携なし。

取り入れるべき： 採用候補 リマインド通知・日付入力サジェスト。通知は「気づかず期限超過」という最も典型的な抜け漏れの直接対策。

立ち位置の再確認：本アプリの土台（インライン Markdown × 多ビュー × 双方向ドラッグ）は競合に対し独自で強い。負けているのは一貫して「入れる速さ（capture）」「集約（aggregate）」「拾い上げる確実さ（agenda/通知）」の3点であり、これはそのまま本プロダクトの哲学（転記レス・脱・散在管理簿・抜け漏れ防止）の未達領域と一致する。

6. ユーザー体験向上の提案（プロダクト哲学に紐づけて）

本アプリの存在理由は 「Markdown を唯一の正とし、① 無駄な転記をなくす ② 散在する管理簿をなくす ③ タスクの抜け漏れを防ぐ」こと。以下の UX 提案はすべてこの3つのゴールのどれに効くかを明示する。各提案は 転記レス 脱・散在 抜け漏れ防止 でタグ付けした。

ゴール A転記をなくす

「見ている場所で直接直す」「一度書けば二度書かない」を徹底する。

A-1 ビュー内インライン編集の全面化

現状ドラッグで日時は書き戻せるが、テキスト・ステータス・優先度・タグは編集できない。アジェンダ/テーブル/カンバンのカード上で全項目をその場編集→Markdown 書き戻しへ。「ビューで気づいた修正をエディタに戻って転記」をゼロにする。

方針：既存 markdown-patch.ts をフィールド単位編集に汎化。(sourcePath,line,field) 指定の最小パッチ。

A-2 貼り付け即変換（他記法→本記法）

ガイド第9章に「プレーンテキスト/Obsidian Tasks/表→本記法」の変換ルールが既に定義済み。これをコード化し、貼り付け時に自動変換すれば、既存メモ・メール・議事録からの取り込み転記が消える。

方針：「変換して貼り付け」コマンド＋ヒューリスティック変換器。日付は chrono-node。

A-3 構造テンプレート / スニペット

定型業務（月次清掃 等）のグループ＋メタ雛形をワンコマンド挿入。毎回の手打ちを排除。

方針：テンプレートを Markdown 断片として保持し挿入。繰り返し（RRULE）と併用。

ゴール B散在をなくす

複数ファイルに散ったタスクを「1か所で見える／寄せられる」状態に。

B-1 常設スマートリスト（サイドバー）

「今日 / 期限切れ / 今週 / 未スケジュール / #タグ別」を常に左サイドに表示。どのノートを開いていても全体像が見える＝「あの管理簿を見に行く」をなくす。

方針：集約インデックス（Phase1）＋保存フィルタを SidebarView に。各行クリックで既存カーソルジャンプ。

B-2 クイックキャプチャ＆インボックス

グローバルホットキーでどこからでも1行追加→ 既定の inbox ファイルへ。「後で清書しよう」と別メモに書いて散らかる事故を防ぐ。後から Refile で定位置へ。

方針：キャプチャモーダル→ 設定の inbox へ追記。NL 日付解釈付き。

B-3 Refile（タスクのファイル間移動）

インボックスや雑多ノートのタスクを、案件ノートへ移動。手でコピペ＝転記・重複の温床を、参照を保ったまま移動に置き換える。

方針：移動元の最小削除＋移動先への挿入を1トランザクションで。sourcePath 前提。

B-4 関連タスク／文脈ナビ

@dependsOn や同タグ・同セクションのタスクへワンクリック移動。文脈を集めて散在を体感的に解消。

方針：安定 ID 確立後、ホバーポップオーバーで関連表示。

ゴール C抜け漏れを防ぐ

「気づかず落とす」を能動的・受動的の両面で潰す。

C-1 抜け漏れ検出パネル（ヘルスチェック）

本提案の目玉。次を一覧で能動表示：① 日付なしの未完タスク ② 過去日付なのに未完（超過）③ [>] のまま N 日放置（停滞）④ 未解決の @dependsOn ⑤ 依存先が完了し「着手可能」になったタスク ⑥ 形式不正で集計から漏れているメタ。「拾えていないタスク」を構造的にあぶり出す。

方針：集約 AST に対するルール群（純関数）＋ HealthView。各ルールは ON/OFF と閾値を設定化。

C-2 記法リント＆自動修正（エディタ内）

ガイドの「よくある変換ミス」表をそのままリアルタイム診断に。@schedule の形式不正は静かにカレンダーから消える＝最も危険な抜け漏れ。下線警告＋クイックフィックスで修正。

方針：CM6 診断（task-decoration.ts 拡張）。パーサーが落とした行を警告として可視化。

C-3 アンビエント警告（件数バッジ＋通知）

リボン/ステータスバーに「超過 N 件・今日 M 件」バッジを常時表示。開かなくても気づける。期限・開始時刻のリマインド通知も。

方針：集約結果のカウントを UI に反映＋常駐タイマー通知。

C-4 ウィークリーレビュー導線

週次で「未スケジュールの backlog に日付を割り当てる」ガイド付きフロー。放置タスクを定期的に棚卸しし、抜け漏れを仕組みで防ぐ（GTD/Sunsama 的レビュー）。

方針：未スケジュール一覧＋デイ・タイムラインへのドラッグ割付（A-1/Day Planner 採用と合流）。

7. ダイレクト操作 ― 「md に書いてから、ドラッグで割り付ける」

設計思想： Markdown は文字しか表現できず、日時・期間・位置・関係といった「非テキストの次元」が苦手である。本アプリの視覚ビュー（カレンダー/ガント/カンバン）は、まさにその次元を表現する平面だ。したがって理想的な操作モデルは ―― 「考えていることは md に文字でバーっと書く → 足りない次元（いつ・どれだけ・どこ）はビュー上のドラッグで直接与える → 即 md に書き戻す」。これにより二重管理が一切発生せず、md だけで完結する。以下の中核機能はユーザー指定の MUST である。

7.0 中核：選択テキスト → ビューへ DnD でメタ付与 MUST

エディタでタスク行（またはその一部テキスト）を選択し、そのままカレンダー/ガント/カンバンへドラッグ＆ドロップすると、着地した場所に応じたメタデータがそのタスクへ書き込まれる。「とりあえず書いて、後でスケジューリング」という自然な思考順序をそのまま操作にできる。

md エディタ ビュー（ドロップ先） md へ書き戻し（自動） ───────────────────────────────────────────────────────────────────────────── - [ ] 見積書を松田さんへ送る ──drag──▶ カレンダー 6/30 14:00 枠 ▶ - @schedule: 2026-06-30T14:00/... - [ ] 議事録の清書 ──drag──▶ カレンダー 終日帯 7/2 ▶ - @due: 2026-07-02 - [ ] 棚卸し作業 ──drag──▶ ガント 7/3〜（バー幅=期間） ▶ - @schedule: 2026-07-03/2026-07-05 - [ ] 稟議承認待ち ──drag──▶ カンバン「対応中」列 ▶ ステータス [>] ＋セクション移動
  • カレンダー：時間グリッドにドロップ→開始時刻つき @schedule／終日帯にドロップ→@due。既定の所要時間（設定）でバー幅を補完。
  • ガント：ドロップ位置＝開始日。横ドラッグの幅で期間（開始/終了）を @schedule に。
  • カンバン：列＝ステータス or セクション or タグ（groupByField に連動）。ドロップで [ ]→[>] 等を書き戻し。
  • 複数行選択で一括割り付け：連続ドロップ／日付を順に配分。「書いたものを一気に予定へ落とす」を高速化。
実装方針： ① エディタをドラッグ源に： CM6 で選択範囲の dragstart を捕捉し、行番号→AST ノードを nodeLineMap の逆引きで解決。DataTransfer に {sourcePath, line, nodeId} を載せる（カスタム MIME）。 ② 各ビューを drop ターゲットに： *ViewMount.svelte に dragover/drop ハンドラを追加し、ドロップ座標→対象の日時/列を算出。 ③ 書き戻し： 既存 markdown-patch.ts を「指定ノードの子リストへ @meta を挿入/更新」操作に汎化（ドラッグ移動の書き戻しと同基盤）。 ④ 誤爆防止： ドラッグ中ゴーストに「何がどこへ書かれるか」をプレビュー（7.1）。対象がタスク行でない場合はドロップ不可表示。

7.1 関連するダイレクト操作の家族（中核と一緒に効く）

★ 未スケジュール・トレイ MUST 相棒

日付の無いタスクだけを集めた引き出し（トレイ）を常設。ここから1件ずつカレンダー/ガントへドラッグして割り付ける。「md にバーっと書く→後で割り付ける」運用そのものを体現し、トレイが空＝全タスクが予定化済みという抜け漏れチェックにもなる。

方針：集約インデックス（Phase1）で @schedule/@due 無しの未完タスクを抽出→ドラッグ可能なリストに。7.0 の drop 基盤を共有。

ドラッグ中の着地プレビュー（ゴースト）

ドロップ前に「@schedule: 6/30 14:00 が書かれます」をその場に表示。文字で結果が見えない md の弱点を補い、誤った割り付けを防ぐ。

方針：dragover で対象スロットを算出し、生成予定の meta 文字列をツールチップ表示。

逆方向：ビューのスロット → md へドラッグ挿入

執筆中に、カレンダーの枠やガントの日付をエディタのカーソル位置へドラッグすると @schedule/@due 行が挿入される。書きながら日付を拾える。

方針：ビュー側を drag 源、エディタを drop 先に。挿入位置はカーソル行の子リスト。

クイック・ドロップ標的（今日/明日/来週チップ）

ビューを開かずとも、サイドの「今日・明日・来週・期限なし解除」チップへドロップするだけで相対日付を付与。最頻ケースを1アクションに。

方針：常設の小型ドロップゾーン。相対日付を絶対日付へ解決して書き戻し。

時間ブロックのペイント

カレンダー上で時間帯をドラッグして「塗る」→そこへタスクをドロップで、その期間を @schedule に。所要時間を意識した予定化（タイムブロッキング）。

方針：範囲選択ジェスチャ＋ドロップを合成。Day Planner 採用（§5）と合流。

ビュー ⇄ エディタ 双方向ハイライト

エディタでタスク行を選択するとそのカードが全ビューで光り、逆も同様。「この行はどこに割り付いているか」が一目で分かり、ドラッグの狙いを定めやすい。

方針：EditorEventBus に hover/selection イベントを追加し各ビューで強調。

7.2 ドラッグに頼らない「自然化」（キーボード／インライン）

ドラッグが最適でない場面（執筆中・キーボード主体・モバイル）でも、同じ「文字を打たずにメタを与える」体験を提供する。

チェックボックス click でステータス循環

エディタ/ビューの [ ] をクリックで [ ]→[>]→[x]→[ ] と循環。マーカーを手打ちしない。低工数・高頻度。

方針：CM6 のチェックボックス装飾にクリックハンドラ（task-decoration.ts 拡張）。

行ホバーのインライン・アクションピル

タスク行にホバーすると行末に 📅 日付 ⚑ 優先 ▸ 状態 のピルが出現。クリックでピッカーが開き、メタを打たずに付与。

方針：CM6 ウィジェット装飾＋ポップオーバー。書き戻しは patch 基盤。

@ オートコンプリート＋日付ピッカー

子リストで @ を打つとメタキー候補、@schedule/@due 選択時はカレンダー・ポップオーバーが開いて正規形を自動挿入。形式ミス（ガイドの注意点表）を根絶。

方針：Obsidian EditorSuggest ＋日付ピッカー。meta-keys.ts を候補源に。

インライン日付チップ（クリックで再編集）

既存の @schedule: … をエディタ上でクリック可能なチップとして描画。クリックでピッカーが開き日時を変更→書き戻し。テキストを手で書き換えない。

方針：CM6 ライブプレビュー装飾。値レンジを保持して置換。

自然言語日付の即正規化

@schedule: 明日10時 や 来週月曜 と打って確定すると 2026-06-29T10:00/… へ自動変換。頭の中の言葉のまま入力できる。

方針：chrono-node（日本語対応）でパースし正規形へ置換。曖昧時は候補提示。

キーボード・スケジュールパレット

タスク行にカーソルを置きホットキー→「today / tomorrow / +1w / 日付指定」パレット。マウスを使わず一瞬で予定化。パワーユーザー向け。

方針：コマンド＋SuggestModal。選択を相対解決して meta 書き戻し。
このセクションの一貫した原理：「md を唯一の正に保ったまま、文字で表現できない次元（時間・期間・位置・関係）を直接操作で与え、即座に md へ書き戻す」。ユーザーは "管理簿を更新する" 意識を持たず、考えを書き、掴んで置くだけでスケジュールが完成する ―― これが本プロダクトが目指す「自然に行わせる」体験の中核である。

8. フィルタ機構の設計指針（ホスト層・常設・パネル非依存）

§4(B)・§5・§6 で挙げたフィルタ／集約を、「パネルごとの内部機能」ではなく「全パネル共通のホスト層機構」として実装するための設計指針。要点は3つ ―― ① データを渡す“境界”で効かせる（パネル非依存）／② エディタは行を消さず“表示しない”（折り畳み）／③ UI は普段最小・必要時展開。

8.1 ホスト層フィルタ ― 「パネルに渡すデータ自体」を絞る

現状、md をパースした階層構造（AST）を各 *ViewMount.svelte がパネル（自作 svelte ライブラリ）へ渡している。フィルタはこの“渡す瞬間”に作用させ、パネルへ渡すデータ自体を絞る。パネル内部の機能（カンバンの groupBy 等）とは完全に独立し、「パネルの実装が何であろうと、共通してパネルへ渡すデータをフィルタする常設機構」として成立する。

┌─────────── ホスト層（エディタ側が提供する親要素）───────────┐ md ──parse──▶ AST(階層) │ [ ▾ フィルタバー（常設・パネル非依存）] │ │ │ filterNodes(doc, query) │ │ ▼ │ │ filtered AST ──▶ Panel（Kanban / Calendar / Gantt …）描画 │ └────────────────────────────────────────────────────────────┘ ▲ パネルは「渡されたデータを描くだけ」＝無改修で共通フィルタの恩恵
  • 設置場所：各 *ViewMount.svelte（＝パネルの親要素）の直下に共通 <FilterBar> を差し込む。パネル本体（ライブラリ）には手を入れない。
  • 作用点：パネルへ props を渡す直前に filteredDoc = filterNodes(doc, query) を挟むだけ。各 Mount は filteredDoc を渡す。
  • 共通基盤：§4(B) の filterNodes、§6(B-1) スマートリスト、§5 採用候補2 の埋め込みクエリブロックと同一のフィルタ実装を共有（条件 DSL も共通）。
  • 階層の扱い：子タスクがヒットする場合は親グループ／セクションを残す（文脈保持）か、ヒット行のみ残すかを設定可能に。
実装方針： 純関数 filterNodes(doc, query): Document（AST→絞り込み済み AST）を新設。共通 Svelte コンポーネント <FilterBar bind:query /> を各 ViewMount に配置。Mount は $: filteredDoc = filterNodes(doc, query) を計算してパネルへ渡すのみ。パネル側は一切変更不要 ＝ 将来パネルが増えても自動で対応。

8.2 エディタへのフィルタ ― 行を消さず「表示しない」（折り畳み）

エディタ側のフィルタは、条件にヒットしない（またはヒットする）タスク行を“非表示”にする。Excel の行の非表示と同じく ―― 行自体は削除せず、表示だけしない（折り畳み）。md 本文は一切変更しないため、「唯一の正」を壊さずに視界だけ整理できる。

  • 非表示＝折り畳み：対象タスクとその子（@meta・> コメント・子タスク）をまとめて畳む。完全に高さ 0 で隠す／薄いプレースホルダで「N 件非表示」を出すかは選択式。
  • 本文不変：テキストは無改変。トグルで即全展開でき、フィルタは“表示レイヤ”に閉じる。
  • 表示／非表示の向き：「ヒットのみ表示」「ヒットを隠す」の両方向を切替（例：完了タスクを隠す／緊急だけ表示）。
実装方針： CodeMirror6 の Decoration.replace({block:true}) もしくは行を display:none 相当にする装飾で、対象行範囲を非表示化（task-decoration.ts と同系統の StateField）。AST の行範囲（lineNumber＋子孫の行域）を filterNodes の判定で得て、非表示レンジ集合を生成。アンフォールド用のガター/コマンドを用意。

8.3 フィルタ UI ― 普段は最小、必要時だけ展開

フィルタ設定エリアは常設だが場所を取らない。普段は小さなアイコン／チップ（適用中の条件数バッジ付き）のみ。クリックで条件入力エリアが展開し、フォーカスアウトで自動的に畳む。

最小表示（既定）: [ ⛃ フィルタ ②] ← 適用中2件のバッジ。クリックで展開 展開時: [ ⛃ ▾ status:!done due:<=今週 +条件 × クリア ]
  • 適用中の明示：条件が効いている間はアイコンをアクティブ色＋件数バッジ表示。「隠れているデータがある」ことを見落とさない（抜け漏れ防止と整合）。
  • 素早い解除：ワンクリックで全解除＝全件表示へ復帰。
  • 一貫した配置：ホスト層（8.1）でもエディタ（8.2）でも同じ最小チップ UI を使い、操作を共通化。

8.4 スコープ：パネル個別 or 全体共通

フィルタ条件はパネルごとに独立（カンバンは未完だけ・ガントは全件、等）を基本としつつ、設定で「全パネル共通の条件に同期」も選べるようにする。共通モードでは 1 か所の変更が全ビューへ波及し、横断的な絞り込み（例：#総務 だけを全ビューで見る）が 1 操作で済む。

フィルタ設計の原理：フィルタはパネルの機能ではなく「md→パネルの間に立つ共通レイヤ」。データ境界で効かせるからパネルは無改修で恩恵を受け、エディタでは“折り畳み”に徹するから md は唯一の正であり続ける。UI を最小常設にすることで、絞り込みが「特別な操作」でなく日常の自然な動作になる。

9. ロードマップ提案

強み（土台）は維持し、競合が標準装備する弱点を順に埋める。フェーズは依存関係順（集約 → その上の機能群）。

Phase 0 — Quick Win（集約基盤を待たず単体で価値）

低リスクで哲学に直接効く。議論後すぐ着手できる候補。

  • 記法リント＆クイックフィックス（C-2）― 形式不正で集計から消える事故を防止
  • 貼り付け即変換（A-2）― ガイド第9章の変換ルールをコード化、取り込み転記を撲滅
  • グループ進捗ロールアップ（派生プロパティのみ・書き戻し不要）

Phase 1 — 集約基盤（土台の拡張）

単一ファイル前提を脱し、ほぼ全機能の前提となるマルチソース化。競合に最も負けている領域の土台。

  • 複数ファイル／フォルダ／Vault スコープの AST インデックス（sourcePath 付与・差分更新）
  • ホスト層フィルタ（§8.1）：filterNodes ＋共通 <FilterBar> を各 ViewMount 直下に。パネル非依存・常設・最小表示 UI（§8.3）
  • エディタの折り畳み式フィルタ（§8.2）：行を消さず非表示（本文不変）
  • ノート埋め込み型ライブ集約ブロック（採用候補2：Dataview/Tasks 相当・同一 filterNodes 基盤）
  • → これが無いとアジェンダ・スマートリスト・抜け漏れ検出がすべて成立しない

Phase 2 — 日常運用（capture / aggregate / agenda）＋ダイレクト操作の中核

競合に負けている「入れる速さ・集約・拾い上げ」を埋めつつ、ユーザー MUST の DnD 割り付けを投入する中核フェーズ。

  • 【MUST】選択テキスト → ビューへ DnD でメタ付与（§7.0）＋着地プレビュー
  • 横断アジェンダ / Today（採用候補1）＋常設スマートリスト（B-1）
  • 抜け漏れ検出パネル（C-1・目玉）＋件数バッジ（C-3）
  • 編集モーダル＋日付ピッカー＋自然言語入力（採用候補3）
  • 繰り返しタスク（RRULE 展開、ビュー側のみ）／リマインド通知
  • @due の終日イベント射影・カレンダーの view 切替

Phase 3 — 転記レスの完成・ダイレクト操作の家族・可視化

  • 未スケジュール・トレイ（§7.1・「書いてから割り付ける」の相棒）＋クイック・ドロップ標的
  • ダイレクト操作の「自然化」群（§7.2）：チェックボックス循環・@補完＋日付ピッカー・インライン日付チップ・自然言語日付
  • 逆方向ドラッグ挿入・時間ブロックのペイント・双方向ハイライト（§7.1）
  • 編集可能テーブル（表）ビュー（採用候補5：Projects 相当）＋ビュー内インライン編集の全面化（A-1）
  • クイックキャプチャ＆インボックス＋Refile（B-2/B-3：採用候補6）
  • デイ・タイムライン（Day Planner 相当）＋ウィークリーレビュー導線（C-4）
  • グループ進捗（Phase0未着手分）／@dependsOn 依存矢印＋安定 ID／色分け／カンバン列＝ステータス遷移／関連タスクナビ（B-4）

Phase 4 — 連携・堅牢性・拡張

  • iCal エクスポート・構造テンプレート（A-3）
  • 外部編集・競合耐性、モバイルフォールバック
  • ステータス/メタキーのカスタマイズ（alias から）

“Quick Win”（小工数・高効果・先行着手候補）

  • グループ進捗ロールアップ — 派生プロパティ追加のみ。書き戻し不要でリスク最小。
  • @due の終日イベント化 — 既存カレンダー変換に射影パスを足すだけで「期限の見える化」が一段進む。
  • クイック追加モーダル — 集約基盤に依存せず単体で価値。記法ミス（ガイドが警告する形式エラー）を構造的に防げる。
