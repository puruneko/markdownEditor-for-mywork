
 

# カレンダーUIライブラリ仕様書

  

# 1. 設計思想（Why）

  

## 1.1 背景

  

既存ライブラリ（例：FullCalendar、SVAR）は以下の問題を抱える：

  

- 利用方法が固定的で拡張困難

- 内部データ構造がブラックボックス

- 「Event」概念に過度に集約されている

- 有償・商用制限が存在するケースがある

- 利用者にデータ管理方法を強要する設計が多い

  

本プロジェクトではこれらを回避し、

  

> **純粋な「描画専用カレンダーUIエンジン」を実装する**

  

ことを目的とする。

  
  

## 1.2 基本理念

  

### ① UIとデータ責務の分離

  

- カレンダーは「描画エンジン」

- データの正本（Single Source of Truth）は選択制

- 双方向データ同期は行わない

- データ変更は必ずイベント経由

  

### ② ドメインの厳密分離

  

「Task」と「Appointment」は絶対に統合しない。

  

### ③ 組み込み前提設計

  

- VSCode拡張

- Obsidian拡張

- Electronアプリ

- Tauriアプリ

- Webアプリ

いずれにも依存なく組み込める設計。

  

### ④ カレンダーライブラリとは

  

- 初めはfullcalendarを目指して作成する（[fullcalendar/fullcalendar: Full-sized drag & drop event calendar in JavaScript](https://github.com/fullcalendar/fullcalendar)）

- fullcalendarのソースコードは必ず確認してください

  

# 2. ライブラリ概要

  

## 2.1 役割

  

入力：

  

```

定義済みデータ構造（Task / Appointment）

```

  

出力：

  

```

レンダリングされたHTML UI

```

  

このライブラリはアプリ全体の一部であり、

  

- 外部UIを書き換えない

- グローバル状態を持たない

- DOM外部に副作用を出さない

  

## 2.2 技術スタック

  

- TypeScript

- Svelte（コンポーネントベース）

- 環境依存なし（Node専用API禁止）

- 日付計算は必ず外部ライブラリ利用（例：Luxon）

  

# 3. ドメインモデル

  

## 3.1 中核概念

  

```

CalendarItem（抽象）

 ├─ Task

 └─ Appointment

```

  

UI上は混在表示可能だが、  

**内部モデルは完全分離する。**

  

## 3.2 データ構造

  

| フィールド  | 型       | 必須 | 説明                              |

| ----------- | -------- | ---- | --------------------------------- |

| id          | string   | ✔    | 一意識別子                        |

| type        | string   | ✔    | 名称                              |

| title       | string   | ✔    | 名称                              |

| start       | datetime | 任意 | 開始予定                          |

| end         | datetime | 任意 | 期限                              |

| status      | enum     | ✔    | `todo / doing / done / undefined` |

| parentId    | string   | 任意 | 親Task                            |

| tags        | string[] | 任意 | 分類                              |

| description | text     | 任意 | 詳細                              |

  

# 4. Task 定義

  

## 4.1 概念

  

成果を前提とする作業単位。  

完了・進捗・責任を持つ。

  

## 4.2 設計上の重要事項

- 時間占有しないTaskも許容

  

# 5. Appointment 定義

  

## 5.1 概念

  

時間拘束そのもの。

成果の有無は問わない。

  

## 5.2 設計上の重要事項

- 完了概念を持たない

  

# 6. なぜ分離するのか（必須理解事項）

  

両者を「Event」として統合すると：

  

- 予定に進捗率を持たせる矛盾

- 完了概念の破綻

- 工数分析不能

- 稼働拘束分析不能

  

したがって：

  

> Task と Appointment は異なるエンティティである。

  

# 7. UI仕様

  

## 7.1 表示モード

  

| 表示  | 軸方向             |

| ----- | ------------------ |

| 1日   | Vertical           |

| 1週間 | Vertical（最優先） |

| 月    | Horizontal         |

| 年    | Horizontal         |

  

初期実装は「1週間表示」。

  

## 7.2 操作性

  

- DnDによる日時移動

- 開始・終了のみ変更可能

- ズームイン / ズームアウト

- tick interval変更可能

- 10,000件表示に耐える

  

描画方式候補：

  

- SVGベース

- 仮想スクロール

- 差分レンダリング

  

## 7.3 デザイン方針

  

- メトロ系配色

- 企業利用可能な控えめデザイン

- 過度な装飾禁止

- 角丸は適度

- スタイルは極力CSSへ委譲

  

# 8. アーキテクチャ

  

## 8.1 データフロー

  

一方向データフロー（Redux思想）

  

```

External Data

   ↓

Calendar

   ↓

Event Dispatch

   ↓

External Handler

   ↓

Data Update

   ↓

Re-render

```

  

---

  

## 8.2 データ管理モード

  

### モードA：外部管理型

  

- データ正本は利用側

- カレンダーは完全ステートレス

  

### モードB：内部管理型

  

- カレンダーが正本保持

- API経由で操作

必ず選択可能にする。

  

## 8.3 イベント設計

  

- すべての操作はイベント経由

- インライン関数禁止

- 登録関数は外部公開

  

例：

  

```

onItemClick

onItemMove

onItemResize

onViewChange

```

  

---

  

## 8.4 カスタマイズ設計

  

- 主要パーツはレンダリング関数として分離

- 上書き可能設計

- スタイル固定値は相対値優先

- CSS変数活用

  

# 9. 非機能要件

  

- 環境依存なし

- グローバル汚染禁止

- DOM直操作禁止

- 日付計算は専用ライブラリ必須

- パフォーマンス劣化時のみ最適化を許可

  
  

# 10. 絶対に守るべき原則

  

1. TaskとAppointmentを統合しな

2. データ正本を強要しない

3. 双方向データ同期をしない

4. 内部ロジックを秘匿しない

5. カスタマイズ可能性を常に残す
