  ---
  良い点

  - コア設計思想がシンプル: 「taskの存在で意味が動的に決まる」という1つのルー
  ルに集約されていて、理解しやすい
  - Markdownネイティブ:
  特別な記法を最小限に抑え、普通のMarkdownとして読み書きできる
  - ブロッククオートのraw扱い: メモ/コメントの逃がし先が明確

  気になる点・改善提案

  1. hasTaskDescendant の定義に矛盾がある

  AST仕様の例（8.2）で、子を持たないリーフタスク（例:
  「機能洗い出し」「ワイヤー作成」）が hasTaskDescendant: true, isGroup: true
   になっています。しかし仕様6.1では:

  hasTaskDescendant = node.type === "task" OR 子ノードにtaskが存在する

  これだとタスク自身は常に hasTaskDescendant: true になり、子がなくても
  isGroup: true になってしまいます。「リーフタスク（実作業）」と「タスクグル
  ープ（作業分解）」の区別がつきません。

  提案: リーフタスクとグループタスクを区別したいなら:
  isGroup = children.length > 0 && hasTaskDescendant
  または hasTaskDescendant
  の定義からタスク自身を除外し、「子孫にtaskがある」のみにする。

  2. ステータス記法の拡張が未定義

  仕様1では [ ], [>], [x] の3種だが、AST仕様では "blocked" | "hold"
  も定義されています。これらのMarkdown記法が未定義です。

  提案: [!] = blocked, [-] = hold など、記法を明示する。

  3. [>] は標準Markdownパーサで認識されない

  MDASTベースのパーサ（remark等）は [ ] と [x] しか認識しません。[>]
  はカスタムパーサが必須になります。

  提案: 実装時にremarkプラグインでの拡張方針を決めておく。または @status:
  doing をメタ情報で表現する代替案も検討。

  4. ID生成の衝突リスク

  id = hash(path.join("/") + text)

  同じパス+同じテキストのノードが複数あると衝突します（例:
  同じカテゴリ下に同名タスクが2つ）。

  提案: 兄弟内のインデックスをパスに含める: 結合テスト/ケースA[0]/正常系[0]

  5. 親子タスクの進捗ロールアップが未定義

  「親タスク =
  タスクグループ」のとき、子タスクが全部完了したら親はどうなるか？自動で done
   にするのか、手動更新なのか。

  提案: ロールアップルール（自動集計 or 手動）を明示する。

  6. メタ情報 @schedule の構文がMarkdownとして曖昧

  @schedule: ... がリストアイテムの子テキストなのか、別のリストアイテムなのか
  、パーサにとって曖昧です。

  提案: インデントレベルやパース方法を仕様で明示する。例えば「タスクの直下に
  インデントされた @key: value 行」というルール。

  7. Section間の関係

  # Webアプリ開発 の下に ## 運用 がある例で、## 運用 は # Webアプリ開発
  のサブセクションなのか独立セクションなのか。現状のAST例では sections
  がフラット配列になっており、見出しレベルの親子関係が失われています。

  提案: セクションもネストするか、parentSectionId を持たせる。

  ---
  全体として

  設計の方向性は筋が良いです。特に動的意味解釈のアイデアはMarkdownの柔軟性を
  活かせています。上記の中で最優先は 1（isGroupの定義矛盾） と 4（ID衝突）
  です。パーサ実装前に解決しておくと手戻りが少なくなります。
