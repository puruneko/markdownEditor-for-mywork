# ライブラリ修正点まとめ

このプロジェクトでは `svelte-calendar-lib` および `svelte-gantt-lib` を `/tmp` 配下の
ローカルリポジトリから `file:` 依存として使用している。
それぞれに対して行った修正・設定変更を以下にまとめる。

---

## svelte-calendar-lib (`/tmp/calendar-for-mywork`)

### ソースコードへの変更

**なし。** ライブラリ本体のソースコードは一切変更していない。

### 統合側の対応（本プロジェクト内）

#### Vite エイリアス設定 (`vite.config.ts`)

```typescript
{
  find: /^svelte-calendar-lib$/,
  replacement: resolve(__dirname, 'node_modules/svelte-calendar-lib/src/index.ts'),
}
```

**理由:**
`dist/index.js` が Svelte 5.51.2 でビルド済みであるのに対し、本プロジェクトの実行時 Svelte は 5.54.1 であり、内部 API 不一致により実行時エラーが発生した。

```
Uncaught TypeError: Cannot read properties of undefined (reading 'call')
```

ソース (`src/index.ts`) を直接コンパイル対象にすることで、Svelte バージョン不一致を回避している。

**注意:** エイリアスは完全一致 (`/^svelte-calendar-lib$/`) にしているため、
CSS のサブパスインポート (`svelte-calendar-lib/dist/index.css`) には影響しない。

#### optimizeDeps 除外 (`vite.config.ts`)

```typescript
optimizeDeps: {
  exclude: ['svelte-calendar-lib'],
}
```

Vite の事前バンドルを無効化し、Svelte コンポーネントとして正しくコンパイルさせるため。

---

## svelte-gantt-lib (`/tmp/ganttchart-for-mywork`)

### ソースコードへの変更

#### 1. `src/utils/timeline-calculations.ts` — バー最小幅の修正

**変更箇所:** `durationToWidth` 関数

```typescript
// 変更前
return Math.max(diffDays * dayWidth, dayWidth) // 最小1セル分の幅

// 変更後
return Math.max(diffDays * dayWidth, 2) // 最小2px（サブデイタスクを正確な幅で表示）
```

**問題:** 最小バー幅が `dayWidth`（= 1日分のピクセル数）に固定されていたため、
1日未満のタスクがすべて同じ幅（24時間相当）で表示されていた。

**修正:** 最小幅を 2px に変更し、サブデイタスクが実際の duration に比例した幅で
表示されるようにした。

#### 2. `src/index.ts` — `getTickDefinitionForScale` のエクスポート追加

```typescript
// 追加
export { getTickDefinitionForScale } from "./utils/zoom-scale"
export type { TickDefinition } from "./utils/zoom-scale"
```

**理由:** `GanttTab.svelte` でズームレベルに対応するマイナーティック間隔を取得し、
ドラッグスナップ単位 (`dragSnapDivision`) を動的に計算するために必要。

この関数は元々ライブラリ内部でのみ使用されていた (`core/gantt-store.ts` 内)。

### 統合側の対応（本プロジェクト内）

#### Vite エイリアス設定 (`vite.config.ts`)

```typescript
{
  find: /^svelte-gantt-lib$/,
  replacement: resolve(__dirname, 'node_modules/svelte-gantt-lib/src/index.ts'),
}
```

**理由:** calendar-for-mywork と同様に、Svelte バージョン不一致対策として
ソース直コンパイル方式を採用。

#### optimizeDeps 除外 (`vite.config.ts`)

```typescript
optimizeDeps: {
  exclude: ['svelte-calendar-lib', 'svelte-gantt-lib'],
}
```

## ライブラリへの修正要望

本プロジェクトの統合過程で判明した問題・改善要望をまとめる。
ライブラリ側で対応された場合、本プロジェクトの回避コードを削除できる。

---

### svelte-calendar-lib への要望

#### [バグ] dist ビルドが Svelte バージョンに強く依存している

**現状:**
`dist/index.js` はビルド時の Svelte バージョンに内部 API が依存しており、
消費側プロジェクトの Svelte バージョンが少しでも異なると実行時エラーになる。

```
Uncaught TypeError: Cannot read properties of undefined (reading 'call')
```

**要望:**
- Svelte の `peerDependency` 範囲を適切に設定する
- または、消費側の Svelte でコンパイルできるよう、`package.json` の `exports` フィールドで `src/index.ts` を `svelte` 条件でエクスポートする

```json
// package.json の理想形
"exports": {
  ".": {
    "svelte": "./src/index.ts",
    "import": "./dist/index.js",
    "types": "./dist/index.d.ts"
  }
}
```

**現状の回避策（本プロジェクト）:** Vite エイリアスでソース直コンパイル

---

### svelte-gantt-lib への要望

#### [バグ] `durationToWidth` の最小バー幅が常に1日固定

**現状:**
```typescript
return Math.max(diffDays * dayWidth, dayWidth) // 最小1セル分の幅
```
「最小1セル」を「1日」と定義しているため、1日未満のタスクは全て同じ幅になる。

**要望:**
最小幅をピクセルベース（例: 2px）にするか、`minBarWidthPx` オプションとして
設定可能にする。

```typescript
// 修正案 A: 固定最小ピクセル
return Math.max(diffDays * dayWidth, 2)

// 修正案 B: 設定可能な最小幅
return Math.max(diffDays * dayWidth, config.minBarWidthPx ?? 2)
```

**現状の回避策（本プロジェクト）:** ライブラリソースを直接パッチ

---

#### [機能不足] `getTickDefinitionForScale` が外部に非公開

**現状:**
`src/utils/zoom-scale.ts` の `getTickDefinitionForScale` は `src/index.ts` からエクスポートされていない。
消費側でドラッグスナップをマイナーティックに合わせるには、この関数が必要。

**要望:**
```typescript
// src/index.ts に追加
export { getTickDefinitionForScale } from "./utils/zoom-scale"
export type { TickDefinition } from "./utils/zoom-scale"
```

**現状の回避策（本プロジェクト）:** ライブラリソースに直接追加

---

#### [機能不足] `onZoomChange` コールバックで `dayWidth` が取得できない

**現状:**
```typescript
onZoomChange?: (zoomLevel: number) => void
```
コールバックは `scale`（浮動小数点のスケール係数）のみを渡す。
消費側で新しい `dayWidth` を計算するには、内部定数 `BASE_DAY_WIDTH = 40` を
知っている必要がある。

**要望:**
```typescript
// 改善案: dayWidth も引数に追加
onZoomChange?: (scale: number, dayWidth: number) => void
```

これにより消費側で `scale * 40` という内部実装に依存した計算が不要になる。

**現状の回避策（本プロジェクト）:** `onZoomChange(scale)` の引数に `scale * 40` を乗算

---

#### [リファクタリング] `GanttConfig.zoomLevel` (1–5) が実際には使われていない

**現状:**
型定義に `zoomLevel?: number` (1–5 の説明付き) が存在するが、
`GanttChart.svelte` 内でこのフィールドは参照されていない。
実際のズーム制御は `dayWidth` で行われる。

**要望:**
- `zoomLevel` を削除して `dayWidth` に一本化する、または
- `zoomLevel` (1–5) から `dayWidth` への変換を `GanttChart` 内部で実装し、
  消費側が `dayWidth` を直接指定しなくて済むようにする

---

#### [機能不足] `dragSnapDivision` がズームに追従しない

**現状:**
`dragSnapDivision` は静的な設定値であり、ズームイン・アウト時に自動で変化しない。
結果として、ズームレベルによってスナップ単位がマイナーティックからずれる。

**要望:**
`snapToMinorTick: boolean` オプションを追加し、`true` のとき自動でマイナーティックに
スナップするよう内部で計算する。

```typescript
interface GanttConfig {
  // 追加提案
  snapToMinorTick?: boolean  // true: マイナーティック単位に自動スナップ
}
```

**現状の回避策（本プロジェクト）:** `onZoomChange` で `dragSnapDivision` を再計算して `config` を更新

---

#### [機能不足] 親ノードの日時範囲が子孫タスクから自動計算されない

**現状:**
`start`/`end` 未設定のノードは「親の開始日 + 1日」になる（`computeNodes` の実装）。
`autoAdjustSectionDates` は存在するが手動呼び出しのみで、初期表示に反映されない。

**要望:**
`computeNodes` 内で `start`/`end` が未設定のノードに対して、
子孫タスクの min(start) / max(end) を自動適用するオプションを追加する。

```typescript
interface GanttConfig {
  // 追加提案
  autoDateFromChildren?: boolean  // true: 未設定ノードを子孫min/maxで自動計算
}
```

**現状の回避策（本プロジェクト）:** `ast-to-gantt.ts` の変換時に子孫 min/max を計算して `start`/`end` に設定

---

## 注意事項

### ライブラリ更新時の対応

`/tmp/calendar-for-mywork` または `/tmp/ganttchart-for-mywork` を
upstream から pull した場合、以下を確認・再適用すること。

| ライブラリ | 確認事項 |
|-----------|---------|
| calendar | Svelte バージョンが一致しているか（エイリアス方式が不要になる可能性） |
| gantt | `durationToWidth` の最小幅が再び `dayWidth` に戻っていないか |
| gantt | `getTickDefinitionForScale` のエクスポートが消えていないか |

### ライブラリのビルド

統合前に各ライブラリをビルドしておく必要がある（`dist/` が存在しないと
`file:` 依存の型解決が失敗する場合がある）。

```bash
cd /tmp/calendar-for-mywork && npm install && npm run build
cd /tmp/ganttchart-for-mywork && npm install && npm run build
```
