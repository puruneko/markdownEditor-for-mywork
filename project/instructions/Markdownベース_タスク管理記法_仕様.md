

# 1. コンセプト

本仕様は、**Markdownをそのまま使ってタスク・構造・メモを一元管理するためのDSL**です。

> **構造は固定せず、タスクの存在によって動的に意味が決まる**

---

# 2. 基本構造

## 2.1 ノード種別

### ■ タスク

```md
- [ ] 未着手      (todo)
- [>] 進行中      (doing)
- [x] 完了        (done)
- [!] ブロック中   (blocked)
- [-] 保留        (hold)
```

> **パーサ互換性に関する注記**
> 標準MDASTパーサ（remark等）は `[ ]` と `[x]` のみ認識する。
> `[>]`, `[!]`, `[-]` はカスタムパーサまたはremarkプラグインでの対応が必要。

---

### ■ リスト

```md
- フェーズ
- カテゴリ
```

---

### ■ メモ（ブロッククオート）

```md
> 補足説明
```

---

### ■ 見出し（Heading）

```md
# プロジェクト
## サブプロジェクト
```

👉 **構造のルート・スコープを分割する役割**

---

# 3. 意味解釈ルール（コア）

## ■ ルール1：taskの存在で意味が決まる

- 子孫にtaskあり かつ 子を持つ → **グループ**

- 子孫にtaskなし → **メモ**

- task自身で子なし → **リーフタスク（実作業）**

> ※ 「子孫にtaskあり」にはノード自身のtask属性を含めない。あくまで子孫の判定のみ。
    

---

## ■ ルール2：構造は動的

```md
- A
  - B
```

↓

```md
- A
  - B
    - [ ] タスク
```

👉 A / B はメモ → グループへ変化

---

## ■ ルール3：ブロッククオートは完全メモ

```md
> - [ ] タスク
```

👉 **タスクとして扱わない（raw扱い）**

---

# 4. グループの種類

|種類|条件|意味|
|---|---|---|
|リストグループ|list + task子孫あり + 子あり|フェーズ・分類|
|タスクグループ|task + task子孫あり + 子あり|作業分解|
|リーフタスク|task + 子なし|実作業|

---

# 5. Headingの役割（重要）

## ■ 役割

Headingは：

> **ツリーの“ルート単位”を分割する境界**

---

## ■ 解釈

```md
# プロジェクトA
- タスク

# プロジェクトB
- タスク
```

👉 内部的には：

```text
Document
├─ ProjectA (root)
│  └─ tasks...
└─ ProjectB (root)
   └─ tasks...
```

---

## ■ ネスト関係

見出しレベル（depth）に基づき親子関係が決まる：

- `## 運用` は直前の `# Webアプリ開発` のサブセクションになる
- 同じdepthの見出しが現れると、前のセクションは閉じる

```md
# A        ← ルート
## B       ← Aの子
## C       ← Aの子（Bと兄弟）
# D        ← ルート（Aとは独立）
```

---

## ■ 特徴

- ガント・カレンダー単位になる

- スコープ分離

- ID空間の分離に使える

- 見出しレベルによる親子関係
    

---

# 6. ネスト構造（MDAST付き）

---

## 6.1 基本例

```md
- フェーズ
  - [ ] タスクA
  - カテゴリ
    - [ ] タスクB
```

---

### ■ MDASTイメージ

```json
{
  "type": "list",
  "children": [
    {
      "type": "listItem",
      "text": "フェーズ",
      "children": [
        {
          "type": "listItem",
          "task": true,
          "checked": false,
          "text": "タスクA"
        },
        {
          "type": "listItem",
          "text": "カテゴリ",
          "children": [
            {
              "type": "listItem",
              "task": true,
              "text": "タスクB"
            }
          ]
        }
      ]
    }
  ]
}
```

---

### ■ 解釈

- フェーズ → リストグループ
    
- カテゴリ → リストグループ
    
- タスクA/B → タスク
    

---

---

## 6.2 タスクのネスト

```md
- [ ] 親タスク
  - [ ] 子タスク
```

---

### ■ MDAST

```json

  "type": "listItem",
  "task": true,
  "text": "親タスク",
  "children": [
    {
      "type": "listItem",
      "task": true,
      "text": "子タスク"
    }
  ]
}
```

---

### ■ 解釈

- 親タスク → タスクグループ
    
- 子タスク → 実作業
    

---

# 7. 実例（完全構造 + MDAST）

---

## ■ Markdown

```md
- 結合テスト

  - テストケースA

      - [ ] シナリオB

  - テストケースB
    - [ ] シナリオテスト
    - [ ] モンキーテスト
```

```json
  "type": "listItem",

  "children": [
    { "task": true, "text": "テスト観点表の再確認" },

    {
      "text": "テストケースA",
      "children": [
        {
          "task": true,
          "text": "シナリオテスト",
          "children": [
            { "task": true, "text": "シナリオA" },
            {
              "text": "シナリオBの準備",
              "children": [
                { "task": true, "text": "サーバ起動" },
                { "task": true, "text": "ログイン" }

          ]
```

# 8. メタ情報（最小方針）

## ■ 方針

> **基本はMarkdownのみで表現する。メタは最終手段**

---

## ■ 使用する場合（例）

```md
- [ ] タスク
  @schedule: 2026-03-01T10:00/12:00
```

---

## ■ 構文ルール

- タスクの**直下行**にインデント付きで `@key: value` 形式で記述する
- メタ行はノードの `children` には含めず、`meta` フィールドに格納される
- 1行につき1つのkey-valueペア

```md
- [ ] タスク
  @schedule: 2026-03-01T10:00/12:00    ← metaとして解釈
  @priority: 1                          ← metaとして解釈
  - [ ] 子タスク                        ← childrenとして解釈
```

## ■ 制約

- 行単位で記述

- keyは固定語彙（schedule, due, priority, dependsOn, tags）

- UIと同期可能

- `@` で始まらない行はメタ情報として扱わない
    

---

## ■ メタを使うべきケース

- 時間（カレンダー連携）
    
- 依存関係
    
- ID（同期）
    

---

# 9. 進捗ロールアップルール

## ■ 方針

> **Markdown上のステータスは手動更新。UI上の集計表示は自動計算。**

## ■ ルール

- タスクグループの進捗率は、子タスクの完了数 / 子タスク総数で自動計算（表示用）
- Markdownファイル上の `[x]` 等のステータスは、ユーザが手動で変更する
- UI側で「子が全完了なら親も完了にする」等の自動更新は**行わない**

## ■ 理由

- Markdownファイルの予期しない自動書き換えを防ぐ
- ユーザの意図を尊重する（子が全完了でも親を完了にしたくないケースがある）
