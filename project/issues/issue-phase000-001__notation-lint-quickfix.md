# 記法リント＆クイックフィックス（エディタ内診断）

## 1. 課題と方針  — 人間が読む

### このissueで解決すること
本アプリの記法（`@schedule` / `@due` など）が形式不正だと、パーサーがその行を**黙って無視**し、カレンダー・ガント等の集計から**静かに消える**。ユーザーが気づけない最も危険な抜け漏れである（例：`@schedule: 2026-06-01T10:00` は開始のみ＝終了 `/...` 欠落で、その予定は一切表示されない）。

この issue では、エディタ上で記法ミスを**リアルタイムに警告表示**し、変換が一意に決まるものは**ワンクリックで正規形へ修正（クイックフィックス）**できるようにする。Markdown 本文はユーザーが適用するまで変更しない。検出対象は `project/knowledge/markdown-notation-guide.md` 第11章「よくある変換ミスと注意点」に準拠する。

### 方針
- 既存パーサーが受理する形を「正」とし、リントの判定をそれに**整合**させる（独自正規表現でパーサーと乖離させない）。
- CodeMirror 6 のリント機構で警告下線とメッセージを出し、一意修正できるものに修正アクションを添える。

### 既存資産の再利用（必読・実装前に読む）
- `src/lib/parser/schedule-normalize.ts` … `@schedule` 正規化の「正」。リントの受理／不受理はこの挙動と一致させる。**新しい判定正規表現を独自に作らない。**
- `src/lib/parser/plugins/remark-meta-fields.ts` … `@キー` の認識範囲（`schedule/due/priority/tags/dependsOn`）。検出対象キーはここに合わせる。
- `src/editor/task-decoration.ts` … CM6 の `ViewPlugin` と `visibleRanges` 走査・「行頭 `>` は対象外」の実装パターン。リントも同じ除外規則を踏襲する。
- `src/settings.ts` の `enableTaskHighlight` … これが OFF のときはリントも無効化する（既存トグルに追従）。
- 新規依存: `@codemirror/lint` を追加し、`linter()` ＋ `Diagnostic`（`actions` で quickfix）で実装する。

### 仕様（確定事項：迷ったらこれに従う）
- 検出する記法ミス（最低限）:
  1. `@schedule` に `/` が無い、または開始・終了どちらか欠落。
  2. `@schedule` / `@due` の日付が ISO（`YYYY-MM-DD` / `YYYY-MM-DDTHH:mm`）でない。
  3. `@meta` がタスク行と同じ行にある（子リスト `- @key:` になっていない）。
  4. 区切りが `/` でない（`〜`・`～`・`から`・全角記号 等）。
- **クイックフィックスは「変換が一意に決まる時だけ」提供**する: 全角コロン→半角、区切り文字 `〜`/`～`→`/`、全角数字→半角 等。**終了時刻が不明な「終了欠落」は警告のみ**（自動補完しない）。
- **誤検出ゼロを最優先**: 正しい行・コードブロック内・引用（行頭 `>`）内は対象外。

### 実装の要点・つまずき
- リントの「正」は必ずパーサー側（schedule-normalize）に合わせる。ここがズレると「警告は出ないのに表示されない」逆転が起きる。
- 検出ロジックは**純関数**（行文字列→診断配列）に切り出し、CM6 配線と分離する。これによりテストが容易になり、CM6 部分の実装は薄くなる。

### 受け入れ基準（すべて満たすこと）
- 終了欠落 `@schedule: 2026-06-01T10:00` に警告が出る（修正アクションは無し）。
- 区切り誤り `@schedule: ...10:00〜11:00` に警告＋`/` への修正アクションが出る。
- 同一行メタ `- [ ] タスク @due: 2026-06-30` に警告が出る。
- 正しい記法・引用内・コードブロック内には警告が出ない。
- 適用するまで本文は不変。
